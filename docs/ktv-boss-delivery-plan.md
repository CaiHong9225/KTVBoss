# K歌打Boss 开发排期

## 目标

用 7 天业余时间完成一个可录视频、可提交代码工程、可现场演示的 demo。

## Day 1：定方案和搭骨架

目标：

- 定最终玩法和页面流程
- 创建 Android 工程
- 跑通首页、战斗页、结算页三页导航

验收：

- 工程可编译启动
- 遥控器可进入挑战
- 页面跳转完整

## Day 2：静态战斗页和 Boss 反馈

目标：

- 完成战斗页主视觉
- 做出 Boss 血条、玩家能量条、连击区、歌词区占位
- 加入基础动画和受击反馈

验收：

- 假数据下 Boss 血量可下降
- 页面视觉已经有战斗感

## Day 3：麦克风接入和实时攻击

目标：

- 接入 `AudioRecord`
- 解析音量数据
- 用音量驱动普通攻击和能量增长

验收：

- 对着麦克风发声时页面会产生即时响应
- Boss 可被真实输入打掉血

## Day 4：连击、高潮段、大招

目标：

- 增加时间轴段落配置
- 做连击和高潮段必杀技
- 补上失败反馈与 Boss 反击

验收：

- 演唱过程具备明显节奏变化
- 至少有一次大招可演示

## Day 5：AI 台词与结果页

目标：

- 接入 AI 文案服务
- 根据战斗数据生成点评、称号和 Boss 台词
- 完善结算页展示

验收：

- 每次挑战结束后可生成差异化文案
- 结果页有记忆点，不像普通分数页

## Day 6：打磨演示效果

目标：

- 调整 UI 配色、字号、动画节奏
- 加入开场和结尾氛围文案
- 修复焦点问题和崩溃问题

验收：

- 远距离看电视也能看懂
- 录屏时没有明显穿帮点

## Day 7：录视频和打包提交

目标：

- 录制 3 分钟演示视频
- 整理代码仓库
- 写投稿说明

验收：

- 视频和代码均可提交
- 一句话简介和酷点表达清晰

## 最低保底版本

如果时间不足，至少保住以下部分：

- 一个可进入的首页
- 一个可发声驱动攻击的战斗页
- 一个会生成结果文案的结算页

只要这三段完整，就已经够参加比赛。

## 风险与规避

### 风险 1：音高检测效果不稳定

规避：

- 优先用音量驱动主反馈
- 把音高稳定度只作为加分项，不作为核心依赖

### 风险 2：AI 接口不稳定

规避：

- 预置一套本地兜底文案模板
- 网络失败时回落本地生成

### 风险 3：UI 工作量过大

规避：

- 优先做一个视觉主题，不做多皮肤
- 把精力集中在战斗页反馈节奏

### 风险 4：临近提交时还在加功能

规避：

- Day 5 之后原则上不再加大功能
- 后两天只做打磨和录制
